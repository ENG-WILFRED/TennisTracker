# Deployment & Cloudflare Configuration

This document summarizes the steps required to deploy **TennisTracker** to a production environment, optimised for performance using Cloudflare.

## 1. Environment Setup

1. **Database**
   - Use Neon Serverless Postgres or another provider.
   - Run migrations:
     ```bash
     npx prisma migrate deploy
     ```
   - Seed initial data (after provisioning the DB):
     ```bash
     npm run seed
     ```
   - Ensure `DATABASE_URL` is set with correct credentials.

2. **Environment Variables**
   Required variables (comma-separated list):
   - `DATABASE_URL`
   - `NEXT_PUBLIC_APP_URL` (your frontend origin)
   - `JWT_SECRET` (used for authentication)
   - `R2_ACCOUNT_ID`, `R2_BUCKET`, `R2_API_TOKEN` (for media uploads)
   - Optional `CLOUDFLARE_API_TOKEN` (if using Workers API for automation)

3. **Install dependencies** and build:
   ```bash
   npm install
   npm run build
   ```

## 2. Cloudflare Configuration

### Workers / Pages

- Deploy the Next.js app to [Cloudflare Pages](https://developers.cloudflare.com/pages/) or Workers.  
  A sample `wrangler.toml` is included in the repo which defines an R2 bucket binding (`MEDIA_BUCKET`) and a Durable Object namespace (`CHAT_ROOMS`).
  Use `wrangler publish` or the Pages CI action to deploy.
- Enable the **Edge Runtime** so that API routes (`/api/*`) run on Workers.
- Bind Cloudflare R2 bucket as an environment variable if running on Workers, e.g. `MY_BUCKET`.  
  The `MEDIA_BUCKET` binding in `wrangler.toml` corresponds to your R2 bucket name.

### R2 (Buckets)

- Create an R2 bucket for storing media (avatars, logos, etc.).
- Create an API token with **Account > Workers R2 Storage** permissions and set it in `R2_API_TOKEN`.
- Example endpoints are handled by `src/lib/media.ts`.

### Cache-control & Caching

- Many API routes already set `Cache-Control: public, max-age=5, s-maxage=5, stale-while-revalidate=10`.
  This instructs Cloudflare to cache responses at the edge for 5 seconds, reducing load on the origin.
- Adjust values according to your traffic patterns; static pages are prerendered automatically.
- Use Cloudflare **Cache Rules** to cache static assets longer (images, JS bundles, etc.).

### Security & DDoS Protection

- Enable the **WAF** (Web Application Firewall) and install Managed Rulesets.
- Create **Rate Limiting** rules for sensitive endpoints (e.g. `/api/auth/login`, `/api/chat/ws`).
- Use **Access Rules** or **IP Firewall** to block abusive traffic.

### WebSockets & Real-time

- The `/api/chat/ws` route uses Cloudflare Workers WebSocket support. For multi-instance scaling
  you should upgrade the in-memory registry to a **Durable Object** or external pub/sub (Redis).
- Ensure the Workers plan supports WebSockets.

### gRPC Service

- A separate gRPC server (see `src/grpc/server.ts`) runs on a normal Node environment (e.g. Cloud Run, DigitalOcean App).
- This server can be used for high‑throughput or binary streaming workloads (analytics, bulk exports).
- Build a container with:
  ```Dockerfile
  FROM node:20-alpine
  WORKDIR /app
  COPY package.json package-lock.json ./
  RUN npm ci --only=production
  COPY . .
  RUN npm run build
  CMD ["node", "dist/grpc/server.js"]
  ```
- Clients can use `src/grpc/client.ts` as an example; update the host/port to the deployed service.

## 3. Performance Tuning

- Add Prisma indexes for common query filters (already added for chat and players).
- Use selective `select` clauses and pagination for list endpoints.
- Offload expensive operations (analytics, reporting) to background jobs or gRPC.

## 4. Media Handling

- Upload user avatars and organization logos via data URIs; the server will push them to R2.
- Public URLs are generated by `getR2Url(key)` in `src/lib/media.ts` and cached by Cloudflare.
- For private media, generate signed URLs using Cloudflare R2 pre‑signed mechanism (not implemented here).

## 5. Running the App

- On Cloudflare Pages, push to the connected Git branch; Pages will build automatically.
- For local testing, run:
  ```bash
  npm run dev
  ```
- To start the gRPC server locally:
  ```bash
  node src/grpc/server.ts
  ```
## Continuous deployment

The repository includes a GitHub Actions workflow (`.github/workflows/deploy-cloudflare.yml`) that builds the project and pushes it to Cloudflare Pages on every `main` branch push.  Set up the following secrets in your repo:

* `CLOUDFLARE_API_TOKEN` – token with Pages and R2 permissions
* `CLOUDFLARE_ACCOUNT_ID` – your account ID

The workflow runs `npm run build` and then invokes the Pages action; it respects the `wrangler.toml` bindings automatically.
## 6. Additional Notes

- Keep `browserslist` up to date with `npx update-browserslist-db@latest`.
- Rotate API tokens and secrets regularly.
- Monitor performance via Cloudflare Analytics and your database metrics.

By following these steps, you can ensure the application runs quickly, scales gracefully, and leverages Cloudflare's edge network for speed and security.
